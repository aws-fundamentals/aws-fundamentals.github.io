1. IAM Basics

AWS IAM (Identity and Access Management) is a service that helps you securely control access to AWS services and resources. It allows you to create and manage AWS users and groups, and use permissions to allow or deny their access to AWS resources.

2. Core Components of IAM

Users

An IAM user is an entity you create to represent a person or service that interacts with AWS resources. Users have various types of credentials:

  • Username: A unique name for the user within the AWS account.
  • Access Credentials:
    • Password: For AWS Management Console access.
    • Access Keys: For programmatic access via AWS CLI, SDKs, or APIs.
    • MFA (Multi-Factor Authentication): Adds an extra layer of security.

Groups

An IAM group is a collection of IAM users. Groups allow you to manage permissions for multiple users at once:

  • Assign policies to groups to control permissions for all users within that group.
  • Users in the group inherit the group’s permissions.

Policies

Policies are JSON documents that define permissions. They specify what actions are allowed or denied on specific AWS resources:

  • Managed Policies:
    • AWS Managed Policies: Policies created and managed by AWS.
    • Customer Managed Policies: Policies created and managed by you.
  • Inline Policies: Policies embedded directly within a user, group, or role, providing a one-to-one relationship.

Roles

An IAM role is an AWS identity with specific permissions. Unlike IAM users, roles are not associated with a single person. Instead, they are meant to be assumed by trusted entities:

  • Service Roles: Allow AWS services to act on your behalf.
  • Cross-Account Roles: Allow users or services in one AWS account to access resources in another account.
  • Temporary Credentials: Allow for temporary access to AWS resources, useful in scenarios like federated user access.

Policies Structure

Policies are defined using JSON. Here’s a basic example of an IAM policy that grants read-only access to S3 buckets:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::example-bucket",
                "arn:aws:s3:::example-bucket/*"
            ]
        }
    ]
}
  • Version: Defines the version of the policy language (typically "2012-10-17").
  • Statement: The main part of the policy, containing:
    • Effect: Whether the policy allows ("Allow") or denies ("Deny") access.
    • Action: The specific actions that are allowed or denied (e.g., s3:ListBucket).
    • Resource: The AWS resources the actions apply to (e.g., an S3 bucket ARN).
    • Condition (Optional): Specifies conditions under which the policy is applied.

3. IAM Best Practices

  • Least Privilege: Grant only the permissions necessary for users to perform their tasks. Avoid using overly broad permissions.
  • Use IAM Roles for Applications: Rather than embedding access keys in your applications, use IAM roles to securely provide permissions.
  • Regularly Review Policies: Regularly audit IAM policies and access controls to ensure they align with your security requirements.
  • Enable MFA: Use multi-factor authentication for IAM users, especially for sensitive operations.
  • Use Groups: Manage permissions through groups rather than assigning policies directly to users.

4. IAM Roles vs. Users

  • Users: Ideal for individuals or services that need long-term access to AWS resources.
  • Roles: Better for temporary access, services, or cross-account access. They help with security by providing temporary credentials and minimizing long-term exposure of access keys.

5. IAM Access Control

  • Permissions Boundaries: Define the maximum permissions a user or role can have.
  • Resource-Based Policies: Policies attached to resources like S3 buckets, which specify what actions users or roles can perform on the resource.
  • Service Control Policies (SCPs): Part of AWS Organizations, SCPs set permission guardrails across multiple AWS accounts in an organization.

6. Federated Access

Allows users from outside AWS (e.g., corporate directory, SAML 2.0, social identity providers) to access AWS resources without needing AWS credentials. Integration with identity providers enables single sign-on (SSO) for users.

7. Monitoring and Logging

  • AWS CloudTrail: Records API calls made in your AWS account, providing logs for auditing and monitoring.
  • AWS Config: Provides a detailed view of the configuration of AWS resources and helps ensure compliance with internal policies.

8. IAM Policies Example

Here are several examples of IAM policies to illustrate different use cases:

8.1. Read-Only Access to S3 Bucket

This policy grants read-only access to a specific S3 bucket and its contents:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::example-bucket",
                "arn:aws:s3:::example-bucket/*"
            ]
        }
    ]
}

8.2. Full Access to EC2 Instances

This policy provides full access to EC2 instances:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": "ec2:*",
            "Resource": "*"
        }
    ]
}

8.3. Restricting Access to Specific Actions on a DynamoDB Table

This policy restricts access to specific actions on a DynamoDB table:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "dynamodb:GetItem",
                "dynamodb:Query"
            ],
            "Resource": "arn:aws:dynamodb:region:account-id:table/ExampleTable"
        }
    ]
}

8.4. Allowing Access Only from a Specific IP Range

This policy allows access to an S3 bucket only from a specific IP range:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": "s3:*",
            "Resource": "arn:aws:s3:::example-bucket/*",
            "Condition": {
                "IpAddress": {
                    "aws:SourceIp": "192.168.1.0/24"
                }
            }
        }
    ]
}

These examples demonstrate various ways to control access using IAM policies. Always ensure that policies follow the principle of least privilege to minimize security risks.

1. IAM মৌলিক বিষয়

AWS IAM (আইডেন্টিটি এবং অ্যাক্সেস ম্যানেজমেন্ট) একটি পরিষেবা যা আপনাকে AWS পরিষেবাগুলির এবং সম্পদের নিরাপদ প্রবেশ নিয়ন্ত্রণে সাহায্য করে। এটি আপনাকে AWS ব্যবহারকারী এবং গ্রুপ তৈরি করতে এবং পরিচালনা করতে দেয়, এবং অনুমতিগুলি ব্যবহার করে তাদের AWS সম্পদে প্রবেশ অনুমোদন বা অস্বীকার করতে দেয়।

2. IAM এর মূল উপাদানগুলি

ব্যবহারকারী

একটি IAM ব্যবহারকারী একটি সত্তা যা আপনি তৈরি করেন AWS সম্পদগুলির সাথে সম্পর্কিত ব্যক্তি বা পরিষেবার প্রতিনিধিত্ব করতে। ব্যবহারকারীদের বিভিন্ন প্রকারের শংসাপত্র থাকে:

  • ব্যবহারকারীর নাম: AWS অ্যাকাউন্টের মধ্যে ব্যবহারকারীর একটি অনন্য নাম।
  • প্রবেশ শংসাপত্র:
    • পাসওয়ার্ড: AWS ম্যানেজমেন্ট কনসোলে প্রবেশের জন্য।
    • এক্সেস কী: AWS CLI, SDK, বা API এর মাধ্যমে প্রোগ্রাম্যাটিক প্রবেশের জন্য।
    • MFA (মাল্টি-ফ্যাক্টর প্রমাণীকরণ): অতিরিক্ত নিরাপত্তার স্তর যোগ করে।

গ্রুপ

একটি IAM গ্রুপ IAM ব্যবহারকারীদের একটি সংগ্রহ। গ্রুপগুলি আপনাকে একসাথে একাধিক ব্যবহারকারীর অনুমতিগুলি পরিচালনা করতে দেয়:

  • গ্রুপগুলিতে নীতিগুলি নিয়োগ করে সমস্ত ব্যবহারকারীর অনুমতিগুলি নিয়ন্ত্রণ করুন।
  • গ্রুপে থাকা ব্যবহারকারীরা গ্রুপের অনুমতিগুলি উত্তরাধিকারসূত্রে গ্রহণ করে।

নীতিসমূহ

নীতিসমূহ হল JSON ডকুমেন্ট যা অনুমতিগুলি সংজ্ঞায়িত করে। এগুলি নির্দিষ্ট AWS সম্পদগুলিতে কোন ক্রিয়াকলাপগুলি অনুমোদিত বা অস্বীকৃত তা উল্লেখ করে:

  • ম্যানেজড পলিসি:
    • AWS ম্যানেজড পলিসি: AWS দ্বারা তৈরি এবং পরিচালিত নীতিসমূহ।
    • গ্রাহক ম্যানেজড পলিসি: আপনি দ্বারা তৈরি এবং পরিচালিত নীতিসমূহ।
  • ইনলাইন পলিসি: একটি ব্যবহারকারী, গ্রুপ বা ভূমিকার মধ্যে সরাসরি এমবেড করা নীতিসমূহ, একটি একে অপরের সম্পর্ক প্রদান করে।

ভূমিকা

একটি IAM ভূমিকা একটি AWS পরিচয় যা নির্দিষ্ট অনুমতিসমূহ সহ। IAM ব্যবহারকারীর বিপরীতে, ভূমিকাগুলি একটি একক ব্যক্তির সাথে যুক্ত নয়। এর পরিবর্তে, এগুলি বিশ্বাসযোগ্য সত্তাগুলির দ্বারা নেওয়ার জন্য উদ্দেশ্যমূলক:

  • সার্ভিস রোল: AWS পরিষেবাগুলিকে আপনার পক্ষ থেকে কাজ করার অনুমতি দেয়।
  • ক্রস-অ্যাকাউন্ট রোল: এক AWS অ্যাকাউন্টে ব্যবহারকারী বা পরিষেবাগুলিকে অন্য অ্যাকাউন্টের সম্পদগুলিতে প্রবেশ করার অনুমতি দেয়।
  • অস্থায়ী শংসাপত্র: AWS সম্পদগুলিতে অস্থায়ী প্রবেশের অনুমতি দেয়, যেমন ফেডারেটেড ব্যবহারকারী প্রবেশের ক্ষেত্রে।

নীতির গঠন

নীতিসমূহ JSON ব্যবহার করে সংজ্ঞায়িত করা হয়। এখানে একটি IAM নীতির একটি মৌলিক উদাহরণ রয়েছে যা S3 বালতিগুলিতে পড়ার জন্য শুধুমাত্র অনুমতি প্রদান করে:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::example-bucket",
                "arn:aws:s3:::example-bucket/*"
            ]
        }
    ]
}
  • Version: নীতি ভাষার সংস্করণ সংজ্ঞায়িত করে (সাধারণত "2012-10-17")।
  • Statement: নীতির প্রধান অংশ, যা অন্তর্ভুক্ত:
    • Effect: নীতি প্রবেশের ("Allow") বা অস্বীকৃতির ("Deny") অনুমতি দেয় কিনা তা উল্লেখ করে।
    • Action: নির্দিষ্ট ক্রিয়াকলাপ যা অনুমোদিত বা অস্বীকৃত (যেমন, s3:ListBucket)।
    • Resource: AWS সম্পদ যা ক্রিয়াকলাপগুলি প্রযোজ্য (যেমন, একটি S3 বালতির ARN)।
    • Condition (অপশনাল): নীতি প্রয়োগের অধীন শর্তগুলি উল্লেখ করে।

3. IAM এর সেরা অনুশীলন

  • সর্বনিম্ন প্রিভিলেজ: ব্যবহারকারীদের তাদের কাজগুলি সম্পাদন করার জন্য প্রয়োজনীয় অনুমতিগুলি প্রদান করুন। খুব বিস্তৃত অনুমতি ব্যবহার এড়ান।
  • অ্যাপ্লিকেশনগুলির জন্য IAM ভূমিকা ব্যবহার করুন: আপনার অ্যাপ্লিকেশনগুলিতে অ্যাক্সেস কী এমবেড করার পরিবর্তে, IAM ভূমিকা ব্যবহার করে নিরাপদে অনুমতি প্রদান করুন।
  • নীতিগুলি নিয়মিত পর্যালোচনা করুন: নিয়মিতভাবে IAM নীতিগুলি এবং প্রবেশ নিয়ন্ত্রণগুলি অডিট করুন যাতে সেগুলি আপনার নিরাপত্তার প্রয়োজনীয়তার সাথে সঙ্গতিপূর্ণ থাকে।
  • MFA সক্রিয় করুন: IAM ব্যবহারকারীদের জন্য মাল্টি-ফ্যাক্টর প্রমাণীকরণ ব্যবহার করুন, বিশেষ করে সংবেদনশীল অপারেশনের জন্য।
  • গ্রুপ ব্যবহার করুন: নীতিগুলি গ্রুপের মাধ্যমে পরিচালনা করুন বরং ব্যবহারকারীদের সরাসরি নীতিগুলি নিয়োগ করুন।

4. IAM ভূমিকা বনাম ব্যবহারকারী

  • ব্যবহারকারী: ব্যক্তিরা বা পরিষেবাগুলির জন্য উপযুক্ত যারা দীর্ঘমেয়াদী AWS সম্পদে প্রবেশের প্রয়োজন।
  • ভূমিকা: অস্থায়ী প্রবেশ, পরিষেবাগুলি, বা ক্রস-অ্যাকাউন্ট প্রবেশের জন্য ভাল। নিরাপত্তার জন্য তারা অস্থায়ী শংসাপত্র প্রদান করে এবং দীর্ঘমেয়াদী এক্সেস কী গুলি কমিয়ে দেয়।

5. IAM প্রবেশ নিয়ন্ত্রণ

  • অনুমতি সীমারেখা: একটি ব্যবহারকারী বা ভূমিকার সর্বাধিক অনুমতি সংজ্ঞায়িত করে।
  • সম্পদ-ভিত্তিক নীতি: সম্পদগুলির মতো যেমন S3 বালতি সংযুক্ত নীতি, যা উল্লেখ করে কোন ব্যবহারকারী বা ভূমিকা সম্পদটির উপর কোন ক্রিয়াকলাপগুলি সম্পাদন করতে পারে।
  • সার্ভিস কন্ট্রোল পলিসি (SCPs): AWS Organizations এর অংশ, SCPs একাধিক AWS অ্যাকাউন্টের মধ্যে অনুমতির রেলগুলি সেট করে।

6. ফেডারেটেড প্রবেশ

বহির্গামী ব্যবহারকারীদের (যেমন, কর্পোরেট ডিরেক্টরি, SAML 2.0, সামাজিক পরিচয় প্রদানকারী) AWS সম্পদে প্রবেশ করার অনুমতি দেয় AWS শংসাপত্র ছাড়াই। পরিচয় প্রদানকারীর সাথে একীকরণ ব্যবহারকারীদের জন্য একক সাইন-অন (SSO) সক্ষম করে।

7. পর্যবেক্ষণ এবং লগিং

  • AWS CloudTrail: আপনার AWS অ্যাকাউন্টে করা API কলগুলি রেকর্ড করে, অডিটিং এবং পর্যবেক্ষণের জন্য লগ সরবরাহ করে।
  • AWS Config: AWS সম্পদের কনফিগারেশনের একটি বিশদ দৃশ্য প্রদান করে এবং অভ্যন্তরীণ নীতির সাথে মেনে চলার ক্ষেত্রে সহায়তা করে।

8. IAM নীতি উদাহরণ

নিচে বিভিন্ন ব্যবহার কেসগুলি প্রদর্শন করতে IAM নীতির কয়েকটি উদাহরণ রয়েছে:

8.1. S3 বালতিতে পড়ার জন্য অনুমতি

এই নীতি একটি নির্দিষ্ট S3 বালতি এবং তার বিষয়বস্তুতে পড়ার জন্য শুধুমাত্র অনুমতি প্রদান করে:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::example-bucket",
                "arn:aws:s3:::example-bucket/*"
            ]
        }
    ]
}

8.2. EC2 ইনস্ট্যান্সে পূর্ণ প্রবেশ

এই নীতি EC2 ইনস্ট্যান্সগুলিতে পূর্ণ প্রবেশ প্রদান করে:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": "ec2:*",
            "Resource": "*"
        }
    ]
}

8.3. একটি DynamoDB টেবিলের নির্দিষ্ট ক্রিয়াকলাপে সীমাবদ্ধ প্রবেশ

এই নীতি একটি DynamoDB টেবিলের নির্দিষ্ট ক্রিয়াকলাপগুলিতে প্রবেশ সীমাবদ্ধ করে:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "dynamodb:GetItem",
                "dynamodb:Query"
            ],
            "Resource": "arn:aws:dynamodb:region:account-id:table/ExampleTable"
        }
    ]
}

8.4. একটি নির্দিষ্ট IP পরিসর থেকে প্রবেশ অনুমতি

এই নীতি একটি S3 বালতিতে শুধুমাত্র একটি নির্দিষ্ট IP পরিসর থেকে প্রবেশ অনুমতি দেয়:

{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": "s3:*",
            "Resource": "arn:aws:s3:::example-bucket/*",
            "Condition": {
                "IpAddress": {
                    "aws:SourceIp": "192.168.1.0/24"
                }
            }
        }
    ]
}

এই উদাহরণগুলি IAM নীতিগুলির ব্যবহার নিয়ন্ত্রণের বিভিন্ন উপায় প্রদর্শন করে। সর্বদা নিশ্চিত করুন যে নীতিগুলি সর্বনিম্ন প্রিভিলেজের নীতি অনুসরণ করে যাতে নিরাপত্তার ঝুঁকি কমানো যায়।
